Introduction

Some reasons why we choose/migrated to git

  • Advantages to individual developer workflows.
  • Advantages to team workflows.
  • Flexibility in workflow. Branches are very seamless.
  • Faster and less error prone than subversion is, no more are files locked, unable to be cleaned or updated.
  • The OSS tools we use all use git and workflows for submitting patches have moved to pull requests.

Mental Model

Stage/Index...

Git is all over the place in what it calls this thing. The most consistent way to think about it is as staging things to be committed.

  • git diff --cached Does a diff of what is staged to be committed vs the HEAD. The documentation tells me that cached is a synonym of staged.
  • index refers to the stage itself
  • using git add to update a file
  • etc... Inconsistencies in documentation.

When talking to others refer to it as the stage or staging area, and staging to be committed.

This is a new feature compared to subversion. The subversion commit was all changes whereas git commit is just staged changes.

Visualizing Commands

Checkout

Switches to a branch or commit by updating the working directory and stage to match that commit, and moving the HEAD.

 
git checkout release
git checkout release ...

Other useful options

  • -m, --merge

    Lets you switch branches when there are modifications in your working directory or stage that would conflict with the branch you are switching to. Normally the command would prevent you from switching in this case.

  • -b <new branch> [<start point>]

    Lets you create a new branch and then instantly checkout that branch.

Visualizing Commands

Branch

Creates a new reference. By default will have that reference point to the current head. Doesn't switch to the new branch.

 
git branch release

Other useful things about git branch

  • <branch-name> [<start-point>]

    Create a branch from any commit

  • no options

    Lists all branches. Include the -a option to list remote tracking branches as well.

  • -m [<branch-name>] <new-branch-name>

    Renames current or given branch name to a new branch name.

  • -d <branch-name>

    Deletes a branch. This only deletes the reference. If no other references exist to the commits reference by the branch then garbage collection will get them eventually.

Visualizing Commands

Merge

Joins two branches/histories together.

 
git merge release

Useful Options

  • --squash

    Has the same affect as the merge except that a commit is not created. So your working directory and stage will have the result of the merge.

  • -s <strategy>, --strategy=<strategy>

    -X <option>, --strategy-option=<option>

    There are a whole bunch of different algorithms that git can use to accomplish the merge. The man page for merge has pretty decent descriptions of what situations they are useful for. For example --strategy=recursive --strategy-option=patience is useful when merging two branches that have diverged significantly.
